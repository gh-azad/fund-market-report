## 1. مسئله‌ی اصلی و چرایی این پروژه

کاربر (ghasem.azad.kntu@gmail.com) یک پروژه‌ی گزارش‌گیری بازار صندوق‌های سرمایه‌گذاری ایران دارد (`fund-market-report`). داشبورد این پروژه با داده‌هایی از سایت‌های ایرانی مثل **فیپ‌ایران (FipIran)** و **کدال (Codal)** به‌روزرسانی می‌شود.

**مشکل قبل از این پروژه:**
- برای جمع‌آوری این داده‌ها، کاربر باید کامپیوتر شخصی/شرکتی‌اش را روشن نگه می‌داشت
- یا خودش دستی اسکریپت‌ها را اجرا می‌کرد
- دسترسی به سایت‌های ایرانی از بیرون ایران (مثل محیط اجرای Claude) معمولاً محدود یا کند است — به همین دلیل داده‌ها باید از داخل ایران گرفته شوند («ایران‌اکسسی»)

**راه‌حل درخواستی:**
یک سرور را در ایران راه‌اندازی کنیم که:
- ۲۴/۷ روشن باشد (نه وابسته به کامپیوتر شخصی کاربر)
- خودش به‌صورت خودکار و زمان‌بندی‌شده داده‌ها را از سایت‌های ایرانی بکشد
- داده‌ها را ذخیره و از طریق یک API در دسترس Claude (در نشست‌های بعدی) قرار دهد
- تا Claude بتواند بدون حضور کاربر، داشبوردها را به‌روزرسانی کند

### 2.3 Relay API — سرویس اصلی جمع‌آوری داده
یک سرویس Node.js روی VPS راه‌اندازی شده که:
- هر روز ساعت **۵:۰۰ بامداد به وقت UTC** به‌صورت خودکار داده‌ها را از فیپ‌ایران می‌کشد (fetch)
- داده‌ها را در یک پایگاه‌داده‌ی **SQLite** روی همان سرور ذخیره می‌کند
- این داده‌ها را از طریق یک **HTTP API محافظت‌شده با توکن** (Bearer token) در آدرس `https://relay.agahreport.ir` در دسترس قرار می‌دهد
- به این ترتیب، Claude در هر نشستی (حتی بدون دسترسی مستقیم به سایت‌های ایرانی) می‌تواند با یک درخواست HTTP ساده و توکن معتبر، آخرین داده‌های جمع‌آوری‌شده را بگیرد

**بررسی وضعیت سرویس (روی خود سرور):**
```bash
sudo systemctl status relay-api
sudo journalctl -u relay-api -n 50 --no-pager
```

### 2.4 مدیریت خودکار با systemd

| واحد systemd | وظیفه | زمان‌بندی |
|---|---|---|
| `relay-api.service` | اجرای دائمی سرویس Relay API | همیشه فعال |
| `relay-fetch.timer` | فچ خودکار داده از فیپ‌ایران | روزانه، ۵:۰۰ UTC |
| `relay-backup.timer` | اجرای بکاپ‌گیری خودکار | روزانه |

**دستورات مفید:**
```bash
systemctl list-timers | grep relay          # دیدن زمان‌بندی تایمرها
sudo systemctl start relay-fetch.service    # اجرای فوری و دستی فچ
sudo systemctl restart relay-api            # ری‌استارت سرویس API (با احتیاط)
```

### 2.5 سیستم بکاپ‌گیری (دوگانه)

**بکاپ محلی (روی خود VPS):**
- اجرا: روزانه
- نگه‌داری: ۳۰ روز آخر
- شامل: فایل پایگاه‌داده SQLite و لاگ‌ها

**بکاپ برون‌سروری (خارج از VPS):**

> ⚠️ مسیر دقیق فایل‌های بکاپ محلی و محتوای فایل سرویس (unit file) در این خلاصه ذخیره نشده. اگر لازم شد، با `sudo systemctl cat relay-backup.service` یا `sudo systemctl cat relay-fetch.service` روی خود سرور تأیید شود.

## 4. کاری که هنوز ناقص است / باقی مانده

### 4.1 انتقال پایپ‌لاین صندوق‌های Multi-NAV (اولویت اصلی)

**وضعیت فعلی این پایپ‌لاین:**
- یک پایپ‌لاین **Python/JavaScript موجود** است که احتمالاً روی **کامپیوتر شرکت** کاربر اجرا می‌شود (نه روی VPS، نه در این ریپو)
- این پایپ‌لاین صندوق‌های چند-NAV را مدیریت می‌کند — مثال: «صندوق صنایع آگاه» که زیرمجموعه‌هایی مثل پترو، اتو، آلیاژ، بانکا دارد
- داده‌ها را **مستقیماً از سایت‌های خود صندوق‌ها** می‌کشد (نه از فیپ‌ایران — این تفاوت مهم است، چون فیپ‌ایران همه‌ی این جزئیات را ندارد)
- خروجی‌های تولیدی: `today.json`، `data.js`، فایل‌های CSV تاریخی
- نتیجه را **مستقیماً به ریپوی گزارش (GitHub) push می‌کند**
- **مهم:** خود اسکریپت‌های این پایپ‌لاین در ریپوی `fund-market-report` نیستند — فقط خروجی‌هایشان (فایل‌های JSON/CSV تولیدشده) در ریپو موجود است

**قدم‌های باقی‌مانده (به ترتیب):**
1. کاربر باید روی کامپیوتر شرکت، اسکریپت‌های اصلی پایپ‌لاین را پیدا کند
2. آن‌ها را در یک ریپوی خصوصی **جدید** GitHub با نام `fund-market-pipeline` قرار دهد (این ریپو هنوز ساخته نشده)
3. یک نشست Claude باید:
   - این ریپو را کلون کند (روی VPS یا محلی)
   - وابستگی‌ها (dependencies) را نصب کند
   - اسکریپت‌ها را تست کند که در محیط VPS هم درست کار می‌کنند
   - یک `systemd timer` جدید بسازد تا این پایپ‌لاین هم روزانه (یا با فرکانس مناسب) خودکار اجرا شود
   - مطمئن شود پایپ‌لاین بعد از اجرا، نتیجه را به ریپوی گزارش push می‌کند (نیاز به GitHub token با دسترسی write)

### 4.2 کدال (Codal)
در گفتگوهای اولیه به کدال هم اشاره شده بود به‌عنوان یکی از منابع داده‌ی هدف، اما تا این لحظه **هیچ اقدام مشخصی برای فچ خودکار از کدال** انجام نشده — این می‌تواند یک آیتم کاری آینده باشد اگر کاربر بخواهد.

## 6. نکات مهم امنیتی و عملیاتی برای هر نشست جدید

- **توکن‌ها را هرگز commit نکن.** توکن API مربوط به Relay و توکن GitHub برای push کردن بکاپ/پایپ‌لاین، باید فقط در فایل‌های env روی خود VPS نگه‌داری شوند (نه در ریپوی گیت‌هاب عمومی یا خصوصی به‌صورت plain text در کد).
- **قبل از ری‌استارت هر سرویس، وضعیت فعلی را بررسی کن** (`systemctl status`, `journalctl`) — چون این سرور بدون نظارت مداوم کاربر کار می‌کند و قطعی سرویس یعنی از دست رفتن یک روز داده.
- **دسترسی SSH فقط با کلید است** — نشست‌های جدید بدون کلید نمی‌توانند به سرور وصل شوند؛ این باید از کاربر خواسته شود.
- **این فایل بر اساس خلاصه‌ی گفتگوهای قبلی بازسازی شده، نه تأیید مستقیم روی سرور در این نشست.** جزئیات ریزی مثل مسیر دقیق فایل‌ها، نام کامل فایل‌های سرویس، و محتوای فایل‌های env باید هنگام اتصال واقعی به VPS دوباره تأیید شوند.

## 7. چک‌لیست شروع سریع (Quick Start) برای هر نشست جدید

```bash
# 1. اتصال به سرور
ssh deploy@194.5.205.63

# 2. بررسی سلامت کلی همه‌ی سرویس‌ها
```
